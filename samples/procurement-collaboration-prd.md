## 2. 背景与问题

公司当前通过邮件、电子表格和即时通信完成采购计划、询价、报价、审批和合同确认。信息分散导致需求版本不一致、供应商报价难以横向比较、审批上下文不完整、合同确认进度不可追踪，也无法形成统一的采购审计记录。

本项目建设一套多应用端采购协同平台。内部采购人员在管理后台组织计划和询价，供应商在门户中报价和确认合同，审批人在移动端处理待办，潜在供应商通过公开网站查看公告并申请入驻。四个应用端共享同一业务状态和数据，但各自保留独立入口与权限边界。

### 2.1 项目综述

采购协同云面向采购需求提出、供应商响应、内部决策、合同确认和公开招募共同参与的采购业务场景，解决邮件、表格与即时通信并行造成的版本分散、报价口径不一致、审批证据缺失和处理状态不可追踪问题。产品以采购计划为业务起点，将询价方案、供应商报价、横向比价、分级审批、合同确认、采购公告和供应商准入组织为一套共享业务数据与状态的协同工作空间。

产品覆盖采购方内部管理、供应商自助响应、移动审批和公开信息服务四类明确入口。不同入口围绕同一采购对象协作，但分别执行角色授权、数据可见范围和操作边界：采购人员组织需求与供应商选择，审批人员基于预算和风险证据作出决策，供应商维护自身报价和合同反馈，访客仅访问公开公告与注册能力。关键写操作、状态迁移、跨端同步和人工决策均需保留可审计记录。

本期范围止于采购协同过程和合同归档，不建设付款执行、电子签章、物流仓储或统一身份系统，也不使用算法替代供应商选择。外部能力只以数据或身份依赖接入，MindFlow 画布需要完整呈现产品信息架构、业务入口、跨端数据流、状态机、异常边界和权限关系。

## 3. 产品目标与非目标

### 3.1 产品目标

- 让采购计划、询价、报价、审批、合同确认和归档形成可追踪闭环。
- 在报价和审批阶段集中展示预算、供应商评分、风险与附件证据。
- 让供应商能够自助接收询价、提交报价、确认合同和查看结果。
- 让每一次状态变化、数据同步和人工操作均有明确发起方、条件和结果。
- 支持按应用端、业务域和角色查看完整跨端业务链路。

### 3.2 非目标

- 本期不实现财务付款执行、电子签章服务和物流仓储系统本身。
- 本期不替代企业统一身份平台，只消费其用户和组织信息。
- 本期不实现智能推荐供应商，仅展示既有评分与规则风险结果。
- 本期不建设供应商移动端；供应商使用响应式 Web 门户。

## 4. 应用端与入口

### 4.1 管理后台

管理后台是采购组织内部规划、寻源、审批跟踪、合同履约和供应商治理的统一工作空间，主要服务采购专员、采购经理、财务审核员和系统管理员。用户从采购工作台进入，以左侧主导航切换业务域，在询价与报价模块内通过父导航项进入二级导航；顶栏提供全局搜索、消息和账户入口。该端负责创建与维护采购侧业务对象、汇总供应商响应、发起内部审批并处理系统治理事务，只展示当前角色被授权的组织、供应商和采购数据。

- 类型：`admin`。
- 用户：采购专员、采购经理、财务审核员、系统管理员。
- 独立入口：采购工作台。
- 主导航：需求与计划、询价与报价、审批跟踪、合同履约、供应商管理、系统设置。
- 页面骨架：后台工作区骨架，包含管理后台顶栏、左侧主导航、内容区和消息抽屉入口。
- 顶栏功能：平台 Logo、平台标题、全局搜索、消息入口、用户头像和账户菜单；账户菜单进入系统设置页。
- 二级导航：后台主导航的“询价与报价”功能项进入询价与报价子导航，子导航包含询价方案和报价对比。

### 4.2 供应商门户

供应商门户是已准入供应商接收采购任务并完成业务响应的自助入口，服务供应商销售和供应商管理员。用户从门户首页查看待报价、合同确认和通知任务，通过任务导航进入报价填写、结果查看、合同确认与企业资料维护。门户只允许访问当前企业被邀请或被授权的业务数据，负责提交供应商侧报价、附件与合同反馈，不提供采购方内部预算、其他供应商明细和审批配置。

- 类型：`web`。
- 用户：供应商销售、供应商管理员。
- 独立入口：供应商门户首页。
- 主导航：询价任务、我的报价、合同确认、企业资料。
- 页面骨架：供应商门户骨架，包含供应商门户顶栏、任务导航、正文和通知中心。
- 顶栏功能：品牌 Logo、门户标题、任务搜索、通知入口和企业账户；企业账户进入企业资料页。

### 4.3 移动审批 App

移动审批 App 是内部审批人在移动场景下处理采购决策的专用入口，服务采购经理、财务审核员和总经理。用户从待办进入审批详情，通过底部导航访问已办、抄送和个人入口；顶栏提供待办搜索、标题和用户入口。该端聚焦查看采购摘要、报价证据、预算风险和审批记录并执行通过、退回或转交，不承担采购计划编辑、供应商报价维护和系统配置。

- 类型：`app`。
- 用户：采购经理、财务审核员、总经理。
- 独立入口：移动审批待办。
- 主导航：待办、已办、抄送、我的。
- 页面骨架：移动审批骨架，包含移动审批顶栏、内容区和底部标签栏。
- 顶栏功能：页面标题、待办搜索和用户头像；用户头像进入“我的”页。

### 4.4 公开采购网站

公开采购网站是企业面向外部发布采购机会和承接潜在供应商准入的公开入口，服务访客供应商与潜在供应商。用户通过公开导航浏览采购公告、查看报名条件、进入供应商注册或帮助中心，页头提供公告搜索和登录/入驻入口，页脚提供政策、联系与帮助信息。该端仅暴露可公开的数据，注册资料提交后进入内部审核链路，不允许访问定向询价、其他供应商报价或内部审批信息。

- 类型：`web`。
- 用户：访客供应商、潜在供应商。
- 独立入口：采购公告列表。
- 主导航：采购公告、供应商入驻、帮助中心。
- 页面骨架：公开网站骨架，包含站点页头、公开导航、正文和站点页脚。
- 页头功能：站点 Logo、站点标题、公告搜索、登录入口和供应商入驻；供应商入驻进入注册页。
- 页脚功能：版权信息、联系方式、政策链接和帮助中心。

## 6. 信息架构与节点类型

以下结构应明确映射到 MindFlow 的五类通用节点。

- `skeleton`：四个应用端各自的页面骨架。
- `navigation`：后台主导航、询价与报价子导航、供应商任务导航、移动底部导航、公开网站导航。
- `page`：工作台、表单、列表、详情、审批、合同、注册、系统设置、企业资料、移动已办、移动抄送、移动我的和帮助中心等完整页面。
- `popup`：发布询价确认、报价撤回确认、审批退回原因、合同修改申请、报价评分编辑。
- `component`：管理后台顶栏、供应商门户顶栏、移动审批顶栏、公开网站页头、公开网站页脚、报价操作、预算风险摘要和审批时间线。

每个应用端卡片使用 `nestedRelation` 连接该端唯一骨架。骨架是布局容器，其布局功能项可以分别连接主导航、底部导航和顶栏、页头、页脚等布局组件；子导航必须由父级导航标题通过 `interaction` 进入，不再由骨架直连。页面的明确区域通过 `nestedRelation` 连接可复用业务组件；打开弹窗属于用户交互，必须从按钮或操作项使用 `interaction` 发起。

### 6.1 骨架布局映射

| 骨架 | 布局功能项 | 目标节点类型 | 目标节点 |
| --- | --- | --- | --- |
| 后台工作区骨架 | 顶栏 | component | 管理后台顶栏 |
| 后台工作区骨架 | 左侧主导航 | navigation | 后台主导航 |
| 供应商门户骨架 | 品牌顶栏 | component | 供应商门户顶栏 |
| 供应商门户骨架 | 任务导航 | navigation | 供应商任务导航 |
| 移动审批骨架 | 移动顶栏 | component | 移动审批顶栏 |
| 移动审批骨架 | 底部导航 | navigation | 移动底部导航 |
| 公开网站骨架 | 站点页头 | component | 公开网站页头 |
| 公开网站骨架 | 公开导航 | navigation | 公开网站导航 |
| 公开网站骨架 | 站点页脚 | component | 公开网站页脚 |

内容区域、正文、通知中心和消息抽屉入口保留为骨架功能项；没有独立结构或明确目标时不创建空组件、不虚构连线。

### 6.2 导航目标映射

- 后台主导航：需求与计划进入采购工作台；询价与报价进入询价与报价子导航；审批跟踪进入审批发起；合同履约进入合同生成；供应商管理进入供应商审核；系统设置进入系统设置页。
- 询价与报价子导航：询价方案进入询价方案编辑；报价对比进入报价对比页。
- 供应商任务导航：询价任务进入供应商门户首页；我的报价进入报价填写；合同确认进入合同确认页；企业资料进入企业资料页。
- 移动底部导航：待办、已办、抄送、我的分别进入对应移动页面。
- 公开网站导航：采购公告进入公告列表；供应商入驻进入注册页；帮助中心进入帮助中心页。

## 7. 核心业务流程

### 7.1 采购计划与询价

1. 采购专员从采购工作台的“新建采购计划”功能项进入采购计划新建页。
2. 新建页的基础信息分组包含计划编号、采购品类、预算金额、期望到货日期和需求部门；附件分组包含需求说明书和预算文件。
3. “保存草稿”将计划数据写入采购计划库并进入采购计划详情页。
4. “提交计划”校验必填项和预算后，系统自动进入询价方案编辑页。
5. 询价方案包含供应商筛选、询价明细和发布设置；“发布询价”先打开发布确认弹窗。
6. 用户确认后生成询价记录，向供应商门户同步定向询价任务，并由系统自动打开询价详情页。

### 7.2 供应商报价与比价

1. 供应商销售从门户首页的待报价任务进入报价填写页。
2. 报价表单包含报价金额、税率、交付周期、联系人、资质文件和报价附件。
3. “保存草稿”仅更新供应商报价草稿；“提交报价”将报价同步到管理后台并显示提交成功状态页。
4. 未超过截止时间时，“撤回报价”打开撤回确认弹窗；确认后回到可编辑状态。
5. 采购专员在报价对比页使用供应商、报价区间、交付周期和评分筛选报价。
6. 报价对比页面的查询列表展示供应商、报价、税率、交期、评分、风险标签和操作栏；操作栏结构连接报价操作组件。
7. 报价操作组件包含查看报价、编辑评分和移出对比；“编辑评分”打开报价评分编辑弹窗。
8. “生成比价报告”写入比价报告数据，并以唯一 `interaction` 进入审批发起页。

### 7.3 审批

1. 审批发起页展示推荐供应商、推荐理由、预算差异、风险提示和审批人配置。
2. “发起审批”创建审批实例，并同步一条移动审批待办。
3. 移动审批 App 收到推送后，系统自动导航到审批详情页。
4. 审批详情包含采购摘要、报价对比、预算风险摘要组件、附件和审批时间线组件。
5. “通过”更新审批状态；全部节点通过后系统自动进入管理后台合同生成页。
6. “退回”打开退回原因弹窗，提交原因后将审批状态改为已退回，并返回询价方案编辑页。
7. “转交”选择新审批人后更新审批任务负责人，页面保持在审批详情。

### 7.4 合同确认与归档

1. 合同生成页包含合同编号、合同金额、付款方式、履约周期、付款条款、违约责任和验收标准。
2. “生成合同”写入合同草案；“发送供应商确认”同步合同任务到供应商门户。
3. 供应商管理员在合同确认页查看条款，可确认、下载或打开合同修改申请弹窗。
4. “申请修改”将修改说明同步回管理后台，合同回到待修改状态。
5. 双方确认后系统自动进入合同归档页，展示合同状态、签署文件、履约计划、付款节点和操作记录。

### 7.5 公开公告与供应商入驻

1. 访客在采购公告列表按品类、地区和截止日期筛选公告，并进入公告详情页。
2. 公告详情展示采购品类、预算范围、报名截止时间和报名条件。
3. 潜在供应商从“申请入驻”进入供应商注册页，填写企业信息、联系人并上传资质文件。
4. “提交注册”写入供应商申请库，并向管理后台供应商审核页同步待审记录。
5. 系统管理员审核通过后，系统创建供应商账号并自动导航到门户登录页；驳回时向申请人发送原因。

## 8. 状态机

### 8.1 采购计划状态组

- 状态组：采购计划状态。
- 状态节点：采购计划详情 · 草稿、采购计划详情 · 待询价、采购计划详情 · 审批中、采购计划详情 · 已通过、采购计划详情 · 已退回。
- 状态变化使用 `statusChange`，所有节点共享同一个非空 `statusGroupId`。
- 采购计划新建页“保存草稿”成功后使用 `autoNavigate` 进入“采购计划详情 · 草稿”，作为状态链入口。
- 草稿提交后变为待询价；发起审批后变为审批中；审批通过后变为已通过；退回后变为已退回。

### 8.2 报价状态组

- 状态组：供应商报价状态。
- 状态节点：报价详情 · 草稿、报价详情 · 已提交、报价详情 · 已撤回。
- 报价填写页“保存报价草稿”成功后使用 `autoNavigate` 进入“报价详情 · 草稿”。
- 保存报价不改变状态；提交、撤回分别触发状态变化。

### 8.3 合同状态组

- 状态组：合同确认状态。
- 状态节点：合同详情 · 草拟中、合同详情 · 待供应商确认、合同详情 · 待修改、合同详情 · 已确认、合同详情 · 已归档。
- 合同生成页“生成合同”成功后使用 `autoNavigate` 进入“合同详情 · 草拟中”。

## 9. 数据流与系统导航

### 9.1 数据流

- 发布询价：管理后台向供应商门户同步询价任务，使用 `dataFlow`。
- 提交报价：供应商门户向管理后台同步报价数据，使用 `dataFlow`。
- 发起审批：管理后台向移动审批 App 同步审批实例和待办，使用 `dataFlow`。
- 发送合同：管理后台向供应商门户同步合同草案，使用 `dataFlow`。
- 提交注册：公开网站向管理后台同步供应商申请，使用 `dataFlow`。

### 9.2 自动导航

- 提交计划成功后系统自动打开询价方案编辑页，使用 `autoNavigate`。
- 移动端收到并打开审批推送后系统进入审批详情页，使用 `autoNavigate`。
- 最后一个审批节点通过后系统打开合同生成页，使用 `autoNavigate`。
- 双方确认合同后系统打开合同归档页，使用 `autoNavigate`。
- 保存采购计划草稿、保存报价草稿、生成合同后分别进入对应初始状态页，使用 `autoNavigate`。

## 10. 异常与边界条件

- 必填字段缺失、预算金额非法或附件上传失败时留在当前表单并展示字段级错误。
- 询价发布时若无可用供应商，发布确认弹窗禁止确认并提示调整筛选条件。
- 报价截止后禁止提交和撤回，已打开页面需要在操作时重新校验截止时间。
- 审批任务已被其他人处理时，当前操作失败并刷新最新状态，不重复写入审批记录。
- 状态变化必须校验当前状态，禁止跨状态跳转。
- 合同已确认后禁止直接修改正文，只能发起补充协议流程，本期仅展示禁用提示。
- 供应商注册的统一社会信用代码重复时阻止提交，并引导联系管理员。

## 11. 非功能要求

- 所有页面、状态、功能分组、功能项和业务连线具有稳定唯一标识。
- 关键写操作需要审计操作人、时间、原状态、新状态和业务对象。
- 列表查询在 10,000 条数据规模下首屏响应不超过 2 秒。
- 文件上传支持 PDF、Word、Excel、PNG 和 JPG，单文件不超过 20 MB。
- 同一业务动作重复提交时不得产生重复计划、报价、审批或合同记录。
- 四个应用端统一展示在一张 MindFlow 画布中，但保留各自独立入口。

## 12. MindFlow 连线规则

- `nestedRelation`：应用端包含唯一骨架；骨架的布局功能项包含导航或布局组件；页面的区域功能项包含业务组件。
- `interaction`：用户点击按钮、菜单、列表行或链接触发的操作或跳转。
- `autoNavigate`：系统在业务成功、推送打开或流程完成后自动切换页面。
- `dataFlow`：跨应用端或跨业务模块的数据写入、传递和同步。
- `statusChange`：同一状态组内的显式状态迁移。
- 除根节点和应用端外，所有 AI/MCP 连线必须从功能组或功能项出口发起；`cardOutletReason` 不能豁免通用节点卡片来源。
- 主导航、底部导航和子导航使用 `navigation`；顶栏、搜索栏、品牌栏、页头和页脚使用 `component`。
- 弹窗由具体按钮或操作项通过 `interaction` 打开，不使用页面卡片级 `nestedRelation`。
- 同一功能组或功能项出口的 `interaction`、`autoNavigate`、`statusChange` 合并计数，最多一条活动边；`nestedRelation` 和 `dataFlow` 允许一对多。
- MCP 正式创作结果中每个活动通用节点必须至少有一条活动入边；人工编辑与保存可暂存未连接节点。

## 13. 验收标准

1. 四个应用端均存在唯一的应用端卡片到骨架入口边，骨架从明确布局功能项连接顶级导航和布局组件；子导航仅由父导航功能项连接。
2. 五种节点类型和五种连线类型均至少存在一个有效实例。
3. 管理后台到供应商门户、移动审批和公开网站的跨端数据链路可被完整追踪。
4. 采购计划、报价和合同三个状态组内的状态节点与迁移合法。
5. 所有活动业务连线均来自功能组或功能项出口，通用节点卡片来源边为零。
6. 所有 MCP 正式节点均有入边；子导航只有父导航一个层级父级；三种受限连线不存在同出口多目标。
7. 按应用端、业务域、角色、页面类型和连线类型查询能够返回一致结果。
8. 每个导航标题均连接明确的下游导航或页面；所有布局组件展示真实的标题、Logo、搜索、账户或页脚功能。
9. 最终 MindFlow 结构校验无错误、无未解释警告，实体统计与分析设计一致。
